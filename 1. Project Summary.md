## 3. Session Presets & Calculation Logic

The dynamic preset engine reconfigures UI inputs, database allocation logic, and mathematical formulations based on the selected use case.

| Preset Type | Entry Logic | Tax / Charge Structures | Split Rule |
|---|---|---|---|
| **Food & Dining** | Item-based entry per individual | SST % + Service Charge % | Proportional Allocation |
| **Grocery Run** | Itemized receipt line items | None (0% service charge) | Equal or Custom Split |
| **Road Trip** | Flat amount entries linked to map waypoints | None | Equal / Custom Split |
| **Long Trip** | Multi-expense collaborative ledger | Invoice / Bill upload | Net-Debt Settlement |
| **Custom** | Manual template overrides | Fully configurable toggles | Configurable |

### üßÆ Expense Split Methods

All presets support three split methods selectable per expense (or per line item within an itemized receipt):

| Method | How It Works | Validation |
|---|---|---|
| **Equal** | Total divided equally among all selected consumers | Automatic ‚Äî no input needed |
| **By Percentage** | Each consumer assigned a custom % share | Percentages must sum to exactly 100% |
| **By Exact Amount** | Each consumer assigned a fixed RM amount | Amounts must sum to exactly total_amount |

> The last consumer in percentage splits absorbs the rounding remainder to guarantee allocations always equal the total.

### üßÆ Mathematical Formulations

#### Proportional Split Formula *(Food & Dining Preset)*

Isolates individual item subtotals and proportionally distributes statutory taxes and fixed service fees:

```
Total_i = Subtotal_i + (Subtotal_i √ó Tax Rate%)
                     + (Subtotal_i / Group Subtotal √ó Flat Fees)
```

#### Net-Debt Settlement *(Long Trip Preset)*

Calculates the absolute net position of each participant to minimize the number of transfers:

```
Balance_i = Total Paid By i ‚àí Total Owed By i
```

- `Balance_i > 0` ‚Üí **Recipient** ‚Äî the group owes this participant money
- `Balance_i < 0` ‚Üí **Debtor** ‚Äî this participant owes money to the group

> The engine uses a **greedy minimization algorithm** to resolve all balances with the fewest possible peer-to-peer transactions.

- Three split methods: Equal, By Percentage, By Exact Amount

- Real-time client-side validation; server-side exception if invalid
- Rounding remainder absorbed by last consumer in percentage mode

## 7. Security Architecture

| Measure | Implementation |
|---|---|
| **Cryptographic Session URLs** | UUIDv4 session strings eliminate sequential ID enumeration attacks |
| **Identity Protection** | Google OAuth 2.0 ‚Äî no local password storage required |
| **Middleware Enforcement** | CakePHP route-level RBAC checks before any controller action |
| **Storage Seclusion** | Cloudinary private-access tokens prevent cross-session receipt leakage |
| **Transactional Reliability** | All balance updates wrapped in `START TRANSACTION / COMMIT` blocks |
| **Device-Aware Routing** | MobileDetect library prevents layout mismatches on mobile browsers |
| **Regulatory Compliance** | Adheres to Malaysia's **PDPA 2010** ‚Äî guests stored only as volatile `guest_name` strings |
